# v0.0.2 / task-claim-rc2

## Objective

Enforce the core orchestration invariant for MCP-driven agent starts:

> An agent slot must not start unless it has claimed exactly one compatible queued task.

## Scope

This release candidate is intentionally narrow. It covers only task claim/start correctness for the MCP `start_agent` and `rerun_agent` control path.

## In scope

- `Invoke-OrchestratorAgentAction.ps1` must call `Claim-OrchestratorQueueTask.ps1` before launching `Start-GmAgentOrchestrator.ps1`.
- Dry-run calls must return claim preview data and must not launch a process.
- Live calls must launch only after `claim.state == "claimed"`.
- No-work states such as `no_queued_tasks`, `no_compatible_task`, and `no_urgent_task` must not launch a process.
- The result JSON must include the claim object and audit path.

## Out of scope

- Dirty-worktree enforcement.
- Repository sync safety changes.
- Patch promotion safety changes.
- Runtime packaging.
- Dashboard redesign.
- PR #38.

## Implementation target

## Required behavior

1. Resolve `scripts/Claim-OrchestratorQueueTask.ps1`.
2. Call it before constructing or starting the agent process.
3. Pass `-Root`, `-SlotName`, `-PassThru`, and `-DryRun:$DryRun`.
4. Store the claim object in the returned JSON as `claim`.
5. If `claim.ok` is false, return structured failure and do not start a process.
6. If `claim.state` is not `claimed` or `claim_dry_run`, return a structured no-work result and do not start a process.
7. If `claim.state` is `claim_dry_run`, return the would-run command and do not start a process.
8. If `claim.state` is `claimed`, start the process as before.

## Acceptance criteria

- No idle worker starts when the queue is empty.
- No idle worker starts when no compatible task exists.
- A live start only occurs after exactly one queue task is claimed into `tasks/active`.
- Repeated starts do not double-claim the same task.
- The result JSON records claim state and audit path.
