## Outreach question bank

### 30-second framing question

> I am building Suzie, a local-first AI orchestration workspace for human-supervised agent work. What is the most important reliability artifact I should build next: evals, source/claim registries, MCP tool contracts, sandboxing, or product demos?

### Technical direction questions

- What is the smallest useful eval suite for an agent system that reads research, edits files, and opens PRs?
- Which agent boundaries should be schema-checked first: task intake, tool invocation, PR creation, queue movement, or final validation?
- What logs would you require before trusting a local orchestrator with semi-autonomous coding work?
- How should Suzie distinguish model error, tool error, retrieval error, stale-source error, quota/capacity error, and operator-policy error?
- Where should Suzie stay intentionally boring instead of adopting a bigger agent framework?

### Security and safety questions

- What are the top prompt-injection risks for research ingestion that later informs coding agents?
- What should be blocked by policy even if a local agent can technically do it?
- How should Suzie handle remote MCP/tunnel endpoints differently from local tools?
- What evidence should be captured before allowing any destructive operation, sync, reset, cleanup, or task movement?
- Which secrets or credentials should never be visible to the model, even in a local-first workflow?

### Career and networking questions

- Which job titles best match this work: AI platform engineer, agentic AI engineer, LLMOps engineer, AI automation architect, AI solutions architect, or AI technical product lead?
- What project artifact would make this credible to hiring teams: demo video, architecture doc, eval report, PR workflow, security threat model, or dashboard screenshot?
- Which communities are most useful for someone building operational agent systems rather than research models?
- What should be cut from the story because it sounds too broad or too speculative?
