This page is a public-safe project wiki profile. It is not a legal biography, medical claim, patent filing, employment verification, or proof of live system state.

## Main Project: Lantern OS

Lantern OS is a staging and release-control repository for a broader local-first workflow. It is currently marked as **pre-v1.0.0 staging** and requires the convergence loop before any artifact is promoted as ready.

Core Lantern OS ideas include:

| Area | Description |
|---|---|
| Convergence Loop | A 12-step process for inspecting repo state, mapping claims to evidence, validating, fixing, and promoting/holding/rejecting artifacts. |
| 12x3 Fleet Contract | A designed 36-slot review ring: 12 convergence steps with one primary role and two backup validators per step. |
| MCP Work Split | A safe split between connector contracts, actual local probes, tool descriptor review, runtime count reports, and held local-only evidence. |
| RAG Dollhouse | A structured memory/intake lane that separates public-safe summaries from private raw inputs. |
| Dashboard Surfaces | Local operator dashboards for project status, wallet/status review, app health, and staged controls. |
| Report Packets | Human-readable reports, outreach packets, and validation summaries with evidence and boundary sections. |

## Working Style

Alex’s project style emphasizes:

1. Inspect before editing.
2. Keep changes small and reviewable.
3. Treat dirty worktrees as high risk.
4. Do not claim readiness without validation.
5. Separate concept, evidence, and private-IP material.
6. Use public-safe summaries when raw source material contains private, sensitive, or proprietary details.
7. Prefer rollback paths, receipts, and explicit held states over confident but unsupported claims.

## Public-Safe Invention / IP Boundary

Some Lantern OS work includes invention candidates, prototype sketches, ring concepts, modular mounting interfaces, and private-IP intake reports. These are handled as **concept candidates** unless and until they are reviewed through a proper private-IP process.

Public-safe language:

- invention candidate;
- private-IP intake;
- public-safe summary;
- concept diagram;
- prior-art review needed;
- no patent status claimed.

Avoid unsupported language:

- patented;
- patent pending;
- proven novel;
- legally protected;
- ready for manufacture;
- clinically validated;
- live fleet proven.
