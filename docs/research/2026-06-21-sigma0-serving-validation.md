### Verdict detail

**1. CONFIRMED** (conf 0.97) — vLLM has MERGED native support for the Ouro looped architecture and can serve Ouro-1.4B-Thinking in a released version.

> Primary-source evidence fully supports the claim, with one decision-critical caveat (adaptive depth) that does not contradict it.

MERGED native support — confirmed via authoritative GitHub API: PR vllm-project/vllm#27794 "[Model][Ouro] Support Ouro Model" returns merged:true, state:closed, merged 2025-10-30 by ByteDance + vLLM maintainers (yinfan.1024@bytedance.com, youkaichao, Jee Jee Li). The merge commit 9956aae adds a NEW first-class model file vllm/model_executor/models/ouro.py (518 lines) — i.e. native in-engine support, not a generic/Transformers-fallback path. It also registers the architecture in vllm/model_executor/models/registry.py as "OuroForCausalLM": ("ouro", "OuroForCausalLM") under _TEXT_GENERATION_MODELS, and updates docs/models/supported_models.md.

Released version — confirmed via GitHub compare API: commit 9956aae is an ancestor of tag v0.11.1 (compare 9956aae...v0.11.1 -> status "ahead", behind_by 0), i.e. it is contained in the v0.11.1 release. It is NOT in v0.11.0 (v0.11.0...9956aae -> "diverged", the commit post-dates the v0.11.0 branch point). The v0.11.1 release notes list "[Model][Ouro] Support Ouro Model (#27794)". So a released, tagged vLLM (v0.11.1, the current stable line; docs.vllm.ai/en/stable/.../models/ouro/ documents it as "Inference-only Ouro model compatible with HuggingFace weights") serves Ouro.

Can serve Ouro-1.4B-Thinking specifically — supported. The vLLM supported-models table lists OuroForCausalLM with examples "ByteDance/Ouro-1.4B, ByteDance/Ouro-2.6B, etc." The -Thinking checkpoints declare the same OuroForCausalLM architecture and load through the identical code path; the ByteDance Ouro project page announced "vLLM and SGLang integration is ready!" (2025-10-30) covering the family including the Thinking variants.

CAVEAT (does not refute the claim, but is decision-critical for the Σ₀ serving decision): vLLM serves the LOOP at a FIXED recurrent depth only. The official HF model card (ByteDance/Ouro-1.4B-Thinking) states "vLLM does not currently support the adaptive exit feature ... the model will always execute the full number of total_ut_steps." So vLLM runs the parameter-shared looped computation (R-step recurrence) but NOT the entropy-regularized adaptive early-exit (Q-exit). For a thin-Windows-client -> Linux vLLM hybrid this means correct looped reasoning at a chosen fixed R, losing only the per-token dynamic-depth speedup — which the local transformers Q-exit path already runs at ~1 s/token anyway. There were also early transformers-version constraints (model card: use transformers<4.56.0; a community KV-cache fix later restored >=4.56 compat), but that pertains to the HF/transformers path, not the vLLM serving path.

**2. REFUTED** (conf 0.78) — SGLang supports Ouro, and its RadixAttention prefix caching works correctly with the looped/recurrent KV state.

> The claim is a conjunction: (A) "SGLang supports Ouro" and (B) "RadixAttention prefix caching works correctly with the looped/recurrent KV state." Both fail under adversarial scrutiny.

(A) is only weakly/aspirationally true. The official Ouro page (ouro-llm.github.io) says "vLLM and SGLang integration is ready! Check out the PR here" but the link points exclusively to vLLM docs (docs.vllm.ai/.../models/ouro). There is a NATIVE vLLM model (vllm/model_executor/models/ouro/) but NO native SGLang model: GitHub code search of sgl-project/sglang returns 0 hits for "ouro" and 0 for "total_ut_steps". The only SGLang primary evidence is PR #28328 / #28322 ("Improve Transformers backend compatibility for trust_remote_code models"), which explicitly names OuroConfig and fixes it so it can LOAD ("Sets pad_token_id from bos_token_id when absent, fixing models like OuroConfig that omit it entirely... unblock several models failing in P1 broad-model accuracy CI"). So Ouro in SGLang = generic HF Transformers fallback backend, and as recently as that PR it was actually FAILING SGLang's CI. The HF model card's SGLang command (python -m sglang.launch_server --model-path ByteDance/Ouro-1.4B) works only because the Transformers backend exists; it is not a looped-aware implementation.

(B) has ZERO supporting primary evidence and strong contrary indicators. No source (Ouro page, HF model card, paper, SGLang docs/code, any PR) states that RadixAttention prefix caching is correct with Ouro's recurrent KV state. The HF model card mentions no RadixAttention caveat at all (it only flags that vLLM disables adaptive exit). Meanwhile the Ouro paper (arXiv 2510.25741) documents that the looped KV state is delicate: during prefill each of the 4 recurrent steps needs its OWN KV cache (reuse degrades GSM8K >10 pts), and naive decode-time reuse can be "catastrophic" (first-step reuse: GSM8K 78.92 -> 18.73). I read SGLang's Transformers backend source (python/sglang/srt/models/transformers.py): it sets _attn_implementation="sglang" and injects one RadixAttention per STATIC layer_idx (attention_instances[module.layer_idx]); it requires _supports_attention_backend=True or raises. Because Ouro reuses the SAME physical layers across recurrent steps, the same layer_idx is invoked multiple times per token with different hidden states, and the radix KV indexing has no recurrent-step dimension. Nothing demonstrates RadixAttention's prefix match/insert/evict semantics are correct (or even produce identical numerics to the reference looped forward) under this reuse. Since the claim asserts correctness as established fact, and the load-bearing half is both unsupported and contradicted by the architecture/paper, the conjunction is refuted as stated. Residual uncertainty (~0.22): SGLang upstream moves fast; it is possible the Transformers fallback happens to thread the loop correctly (HF's own modeling code drives the loop and SGLang just provides per-layer attention/KV), in which case "supports + works" could be quietly true — but no primary source verifies this, so the strong factual claim cannot stand.

Caveat on the decision context: for Σ₀=Ouro, the SAFE engineering read is vLLM (native model executor) over SGLang. Even vLLM forces full total_ut_steps (no adaptive Q-exit), which matters because the project's whole differentiator is the adaptive-depth loop. Do NOT assume SGLang gives you correct prefix caching over the loop for free.
> Sources: https://ouro-llm.github.io/, https://huggingface.co/ByteDance/Ouro-1.4B, https://huggingface.co/ByteDance/Ouro-1.4B/blob/main/README.md, https://github.com/sgl-project/sglang/pull/28328, https://github.com/sgl-project/sglang/pull/28322, https://github.com/sgl-project/sglang/blob/main/python/sglang/srt/models/transformers.py

**3. REFUTED** (conf 0.9) — At least one of vLLM or SGLang PRESERVES Ouro's adaptive/entropy early-exit rather than pinning a fixed recurrent depth (R4).

> Primary sources contradict the claim for BOTH engines.

(1) vLLM — explicitly does NOT preserve adaptive exit. The official ByteDance Hugging Face model cards (Ouro-1.4B, Ouro-1.4B-Thinking, Ouro-2.6B) all state verbatim: "vLLM does not currently support the adaptive exit feature due to its inference optimization characteristics. When using vLLM, the model will always execute the full number of total_ut_steps." So under vLLM you get a FIXED recurrent depth (the configured total_ut_steps, e.g. R4), not the entropy/Q-exit adaptive depth. early_exit_threshold (default 1.0) and total_ut_steps (default 4) are documented config knobs, but the model card says the adaptive-exit knob is inert under vLLM.

(2) SGLang — no source claims it preserves adaptive exit, and the paper groups it WITH vLLM as a fixed-path engine. The Ouro paper (arXiv 2510.25741, "Scaling Latent Reasoning via Looped Language Models", Section 4.5 / RL Attempts) states: "vLLM/SGLang provide fast rollouts via a fixed execution path, which breaks under LoopLM's variable-depth computation." The authors describe being forced to generate full four-step rollouts and pick the first token exceeding the termination threshold to *simulate* an early exit, producing an off-policy mismatch (tokens produced at final depth, losses at earlier depth) that "did not improve performance" — i.e. they could NOT run true variable-depth through these engines. The HF cards list SGLang only as a deployment option (install/serve instructions) with no capability carve-out vs vLLM and no statement that it honors early_exit_threshold.

(3) Architectural mechanism corroborates: both vLLM and SGLang compile a static/fixed execution path optimized for throughput and batched rollouts; data-dependent per-token recurrent depth (the entropy-regularized Q-exit where a token exits at the first step with CDF(t) >= q) requires a variable trip count that these fixed-path engines do not execute. This matches the project's own note that the looped arch is "runtime locked to transformers 4.57" with "no llama.cpp."

(4) Adversarial counter-search for any vLLM/SGLang PR, doc update, or model-card revision (through June 2026) claiming adaptive-exit/variable-depth support for the looped arch found NONE.

Net: neither engine is documented to preserve adaptive/entropy early-exit; vLLM is explicitly stated to pin fixed total_ut_steps, and SGLang is bundled with vLLM as a fixed-execution-path runtime in the paper. Residual uncertainty: SGLang's silence is technically weaker negative evidence than vLLM's explicit disclaimer (its card never says "SGLang lacks adaptive exit" in those words), but the paper's joint "vLLM/SGLang ... fixed execution path" statement plus the total absence of any positive adaptive-exit claim makes the refutation solid rather than merely uncertain. Decision implication for the repo: Option 3/4 (vLLM/SGLang sidecar or cloud) buys speed but DROPS adaptive depth, collapsing Σ₀ to a fixed-R loop — the same loss-of-loop tradeoff feared for the dense fallback, except the param-shared depth is still computed (just non-adaptively). To keep the adaptive Q-exit you need either the native transformers loop (current ouro_serve.py OURO_NATIVE path) or a custom graph that expresses a data-dependent trip count (the Option 5 ONNX Loop/Scan route), NOT stock vLLM/SGLang.
> Sources: https://huggingface.co/ByteDance/Ouro-1.4B (model card: 'vLLM does not currently support the adaptive exit feature ... the model will always execute the full number of total_ut_steps'), https://huggingface.co/ByteDance/Ouro-1.4B-Thinking (same vLLM limitation; SGLang silent on adaptive exit), https://huggingface.co/ByteDance/Ouro-2.6B/blob/main/README.md (repeats vLLM limitation; lists SGLang only as a deployment option, no adaptive-exit claim), https://arxiv.org/html/2510.25741v2 (Scaling Latent Reasoning via Looped Language Models, Sec 4.5: 'vLLM/SGLang provide fast rollouts via a fixed execution path, which breaks under LoopLM's variable-depth computation'; off-policy mismatch from simulating early exit on full 4-step rollouts), https://arxiv.org/pdf/2510.25741 (paper PDF, same fixed-execution-path statement), https://ouro-llm.github.io/ ('vLLM and SGLang integration is ready! Check out the PR' — announcement only, no adaptive-exit capability claim)

**4. CONFIRMED** (conf 0.97) — ONNX Runtime's Loop operator supports a DATA-DEPENDENT trip count / early termination condition, so adaptive-depth recurrence is expressible in an exported ONNX graph.

> I attempted to refute this and could not — the evidence is decisive at BOTH the spec and runtime-implementation levels, from three independent primary sources.

(1) ONNX OPERATOR SPEC (onnx.ai operator reference, opset-current): The Loop operator's verbatim operating-mode pseudocode includes a pure while-loop with NO trip count — `input ("", cond): for (int i=0; cond; ++i) { cond = ...; }` — and a combined mode `input (trip_count, cond): for (int i=0; i < trip_count && cond; ++i) { cond = ...; }` that terminates EARLY when cond goes false before M iterations. The spec documents both inputs as optional: M = "A maximum trip-count for the loop specified at runtime. Optional. Pass empty string to skip"; cond = "A boolean termination condition. Optional. Pass empty string to skip." The condition is explicitly "a loop-carried dependency for the body graph," and the body re-emits it: body has "1+N+K outputs: (condition, loop carried dependencies..., scan_outputs...)." So the trip count is genuinely data-dependent (the subgraph recomputes the boolean each step). This is the textbook construct for adaptive-depth / variable-iteration recurrence.

(2) ONNX SOURCE (onnx/defs/controlflow/defs.cc): Confirms the body signature emits (condition, loop-carried deps, scan_outputs), i.e. a fresh condition per iteration.

(3) ONNX RUNTIME IMPLEMENTATION (microsoft/onnxruntime .../cpu/controlflow/loop.cc): The runtime ACTUALLY implements early termination, not just specifies it. The execution loop is `while (iter_num_value < max_trip_count_ && *condition_mlvalue_.GetMutable<Tensor>()->MutableData<bool>())`, and each iteration sets `condition_mlvalue_ = fetches[0]` — reading the boolean from the subgraph's first output. When no M tensor is provided, `max_trip_count_` defaults to INT64_MAX, so condition-only (while-style) loops run. This directly addresses the claim's wording ('ONNX RUNTIME's Loop operator supports...').

CAVEAT (bounds the claim, does not refute it): The claim as written is about EXPRESSIBILITY of the control structure in an exported ONNX graph — that is fully confirmed. It does NOT establish two adjacent things the parent COMPILE-AND-EMBED option still owns as unverified risk: (a) that the DirectML execution provider (vs the CPU EP quoted above) honors data-dependent Loop/early-exit on GPU — EP coverage of control-flow ops varies and I did not find primary evidence either way in this pass; and (b) that Ouro's specific entropy-regularized adaptive Q-exit survives torch.onnx.export into a Loop subgraph (PyTorch tracing of data-dependent Python control flow is the classic export pain point, typically requiring TorchScript scripting or torch.cond/torch.export rather than naive tracing). Those are separate engineering claims outside this one's scope.
> Sources: https://onnx.ai/onnx/operators/onnx__Loop.html, https://github.com/onnx/onnx/blob/main/onnx/defs/controlflow/defs.cc, https://github.com/microsoft/onnxruntime/blob/main/onnxruntime/core/providers/cpu/controlflow/loop.cc, https://github.com/onnx/onnx/commit/7abd834091f1024c11749dcfd25126802db9fdd5

**6. PARTIALLY-TRUE** (conf 0.86) — A weight-tied looped transformer (Ouro) has a VRAM footprint of roughly its base parameter count (~1.4B) regardless of the number of loop iterations; recurrence costs compute/time, not memory.

> PRIMARY EVIDENCE from the Ouro paper itself (arXiv 2510.25741, ByteDance Seed et al.) and a dedicated follow-up (MELT, arXiv 2605.07721) both contradict the absolute 'memory not affected by loops' framing.

PART 1 — weights constant: CONFIRMED. Ouro is 'a group of weight-tied layers iteratively reused during the forward pass'; the 1.4B model stays 1.4B params at R=1, R=4, or extrapolated depth. So model-weight VRAM is genuinely depth-independent. This is the true kernel of the claim and the reason a looped 1.4B is light on weights vs an 8-12B dense model it matches.

PART 2 — total memory / KV cache: REFUTED as stated. The Ouro paper: 'each recurrent step requires maintaining its own KV cache, leading to 4x memory overhead for our 4-step model.' During PREFILL all 4 recurrent passes need separate caches because each pass produces a semantically distinct latent state; forcing a shared cache costs >10 GSM8K points. So memory IS coupled to loop depth in the baseline. The MELT paper makes this explicit and quantified: baseline 'per-iteration' KV cache gives memory ~ O(N x L x T) (T = recurrent steps), which MELT reduces to O(N x L) by sharing one KV cache per layer across loops — empirically 25.17 GB -> 6.29 GB (~4x) for 32k-token generation. The very existence of a paper titled 'Decoupling Compute from Memory in Looped Language Models' proves that in the baseline, compute and memory are NOT decoupled — i.e., the claim's second half is the thing researchers had to FIX, not a property you get for free.

MITIGATION the claim ignores: Ouro DOES offer KV-cache reuse for the DECODE phase only (last-step-only or averaged): ~4x KV reduction with ~0.07-0.19 GSM8K-point loss. With that applied at decode, the practical decode-time footprint approaches a standard same-param model. But prefill overhead remains.

NET: weights ~ base param count (true); 'recurrence costs no memory' (false in general — KV cache grows linearly with depth). Hence partially-true with a material correction. Confidence 0.86: two independent primary sources (the model's own paper + a peer follow-up) agree on the O(N x L x T) KV scaling and the 25.17->6.29 GB figure; some numbers were read via WebFetch summaries of the arXiv HTML rather than verbatim from the PDF, hence not 0.95.
> Sources: https://arxiv.org/abs/2510.25741, https://arxiv.org/html/2510.25741v2, https://arxiv.org/pdf/2510.25741, https://arxiv.org/html/2605.07721v1, https://huggingface.co/ByteDance/Ouro-1.4B, https://ouro-llm.github.io/

**7. PARTIALLY-TRUE** (conf 0.82) — llama.cpp / GGUF cannot run Ouro's weight-tied recurrent-DEPTH architecture without custom code changes (its Mamba/RWKV support is recurrent STATE, which is different).

> CORE CLAIM (custom code required): CONFIRMED by primary sources. (1) Ouro-1.4B/2.6B is a parameter-shared looped transformer that re-applies the same layers over depth, controlled by config fields total_ut_steps (default 4) and early_exit_threshold; it requires trust_remote_code=True and pins transformers<4.56.0 — i.e., a genuinely non-standard architecture with custom modeling code (HF model card). (2) llama.cpp's own guides state that adding ANY new architecture requires BOTH a Python converter class in convert_hf_to_gguf.py AND new C++ in src/llama-arch.h/.cpp (new llm_arch enum + LLM_ARCH_NAMES entry) plus a graph builder in src/llama-model.cpp; configuration/metadata alone is explicitly insufficient (HOWTO-add-model.md; discussion #16770, which also surfaced 'you cannot perform loops... you will have to do it inside a custom operation'). (3) No Ouro/OuroForCausalLM arch is registered in llama.cpp's converter (no evidence found). (4) The closest public analog — HRMTextForCausalLM, another recurrent-DEPTH reasoning model — is NOT officially supported and required a third-party patch + custom GGUF conversion, with the contributor noting the novelty is 'entirely in the outer recurrence loop' (discussion #23415). So Ouro cannot run as-is; code changes are mandatory. PARENTHETICAL (Mamba/RWKV = recurrent STATE, 'which is different'): CONFIRMED at the mechanism level — llama.cpp's recurrent support uses llama_memory_recurrent with R State / S State tensors holding SSM/RWKV hidden state recurring over the SEQUENCE dimension, which is categorically different from re-applying a weight-tied block over DEPTH (DeepWiki memory-recurrent / supported-architectures). HOWEVER, the parenthetical's implication that this difference makes the existing infra irrelevant is slightly overstated: the HRM discussion frames adding recurrent-depth as 'low-risk' precisely BECAUSE llama.cpp already handles recurrence for RWKV/Mamba, treating it as a useful scaffold. Additional nuance: a fixed default total_ut_steps loop is config-driven (not tensor-dependent) and could likely be statically UNROLLED in the C++ graph builder without a bespoke GGML loop op; only Ouro's ADAPTIVE early-exit (data-dependent trip count) would truly require custom control-flow ops. ### MLC-LLM / Apache TVM / WebLLM as a native-Windows + WebGPU portable runtime for Σ₀ (option 5 "compile-and-embed" lateral path, MLC variant)

MLC-LLM is the strongest single-codebase fit for the "embed Σ₀ in a Windows client, all-GPU-vendor, no server" goal — and it is the only mainstream engine that ALSO gives you in-browser WebGPU for free. It compiles models to native Windows via Vulkan (.dll) covering NVIDIA/AMD/Intel, and to WebGPU+WASM for browsers, from ONE model definition. Models are defined in a PyTorch-like `tvm.relax.frontend.nn.Module` (pure Python), and crucially the per-layer stack is a normal Python for-loop that is traced/unrolled into the Relax graph — so a weight-tied/parameter-shared looped (Universal-Transformer / Ouro) arch is naturally expressible by reusing ONE layer object N times in that loop, with `tie_word_embeddings` already a first-class config. A 1.4B model at 4-bit needs ~0.9-1.5 GB VRAM and runs in-browser today (WebLLM ships Llama-3.2-1B at 879 MB q4f16 / 1129 MB q4f32), comfortably inside the 8GB dev box and most consumer GPUs. THE catch: this is the SAME custom-arch + adaptive-exit porting risk as the ONNX path, plus MLC has a smaller ecosystem; and Ouro's *data-dependent* early-exit (variable recurrent depth per token) is the genuinely hard part — a *fixed* recurrent depth (e.g. R4) is a trivial static unroll, but runtime per-token Q-exit needs a dynamic loop with a data-dependent trip count, which is engineering work in both MLC/Relax and ONNX. Versus ONNX/DirectML: ONNX Runtime + DirectML (the Phi-3 / Windows-ML path Microsoft ships) is more mature on Windows, has Microsoft backing, and supports DirectX12 across all vendors, but it has NO browser story and its GenAI model-builder is a fixed allow-list (Llama/Phi/Qwen/Gemma/...) that does not include a looped arch — you'd hand-author the ONNX graph (incl. an ONNX Loop op for recurrence) yourself. Net: MLC wins on portability+loop-expressiveness+browser reach; ONNX/DirectML wins on Windows maturity+vendor support. Both are real-but-unverified engineering for the adaptive-exit loop specifically.

### llama.cpp / GGUF / Ollama for serving Ouro's weight-tied recurrent-depth loop (Option 2 feasibility + an unroll shortcut)

VALIDATED (high confidence): stock llama.cpp/GGUF/Ollama cannot run Ouro's weight-tied, adaptive-DEPTH loop without custom C++ changes. GGUF is fundamentally a per-block (blk.N) container with a fixed integer n_layer/block_count and a statically-built forward graph; it has no primitive for weight-tying across layers nor for a data-dependent number of layer applications. llama.cpp's existing 'recurrent' support (Mamba, RWKV) is recurrence over SEQUENCE STATE (a hidden state carried token-to-token, stored via llama_memory_recurrent / cache_r_l, cache_s_l), which is a DIFFERENT axis from Ouro's recurrence over DEPTH (re-applying the same block to refine one token's representation). Supporting Mamba/RWKV gives llama.cpp zero machinery toward Ouro's loop. No fork/issue/PR adding universal-transformer / looped-depth / weight-tied-layer support to llama.cpp was found in searches (searched convert_hf_to_gguf, HOWTO-add-model, ALBERT weight-sharing, universal/looped/depth-recurrent transformer). UNROLL SHORTCUT IS VIABLE as a pragmatic path: because Ouro applies an IDENTICAL block with NO per-step conditioning (paper §3: F^(t) = lmhead ∘ (M^L)^t ∘ emb; no step embeddings / timestep / per-iter norms — confirmed against the paper and the local loop_lm.py), the conversion script can DUPLICATE the one shared block's tensors into N=4 sequential blk.0..blk.3 and emit a vanilla N-layer GGUF that runs unmodified on llama.cpp/Ollama/all hardware. WHAT IS LOST: (1) adaptive depth / Q-exit — you bake a FIXED depth (e.g. R4) for every token, forfeiting the per-token early-exit that is Σ₀'s whole 'adaptive computation' thesis; (2) the convergence/fixed-point exit mode; (3) the exit-gate telemetry the product surfaces (mean_depth, canary). So unroll = a fast, all-hardware, normie-friendly DENSE approximation of Σ₀ that DROPS adaptive depth and the memory win — functionally it collapses Option 2's 'drops the loop' problem in a different guise.
