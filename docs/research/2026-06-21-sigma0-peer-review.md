# Σ₀ Portfolio + Deck — Peer Review

## Lens: Technical correctness & staff-level rigor: are architecture claims grounded in the verdicts (not overclaimed)? Unsupported statements, hand-waved perf, missing failure modes/rollback? Is the ONNX-export + adaptive-exit risk honest? Is the 8GB / cold-start / latency math credible?
- **Grade:** B  ·  **Staff-level:** True  ·  **Verdict:** revise

**Strengths:**
- Architecture claims are unusually well-grounded in the verdict layer. The two refutations that matter most (vLLM/SGLang do NOT preserve adaptive Q-exit, conf 0.90; SGLang RadixAttention-over-loop unverified/suspect, conf 0.78) are correctly load-bearing in the design: vLLM is demoted to an opt-in fixed-R4 'fast' tier rather than the default. This is exactly the right inference and the portfolio doc states it without overclaiming.
- The ONNX-export bet is honestly gated. The doc repeatedly and correctly distinguishes the CONFIRMED ONNX Loop primitive (data-dependent trip count, conf 0.97) from the UNPRECEDENTED Ouro-specific export, names the three sub-risks (weight-tie dedup, DirectML GPU-EP control-flow coverage, int4 quality on a looped model), and makes NO-GO fall back to cloud-transformers rather than a loop-dropping dense GGUF. The 'CPU EP confirmed, GPU EP unverified' caveat is preserved faithfully from the verdict and elevated to an open question. This is staff-level intellectual honesty about an unproven path.
- The swappable-client-contract framing (one SIGMA0_BASE_URL indirection, engine-as-config, no per-backend branching) is a genuinely sound architecture that honors the 'models are replaceable' North Star and avoids the dense-coder-track sprawl the consolidation forbids. It is verified against the actual repo (stream-chat.js reads OLLAMA_BASE_URL at :1270, ouro_serve.py is an Ollama shim) — not hand-waved.
- The QLoRA-vs-'persistent-learning-not-weight-modification' tension is explicitly named rather than buried, and the reconciliation (one-time, human-triggered, eval-gated format-alignment on a frozen base; experiential learning stays in memory+retrieval) is defensible and correctly scoped.
- Phasing puts all unproven research (Phase 3 ONNX spike) off the critical path behind a low-risk shipping floor (Phases 0-1 cloud-transformers), with explicit GO/NO-GO and parity gates at every boundary. The memory/KV-cache nuance (weights depth-flat ~2.8GB; KV ~4x at R4 only on long contexts) is carried correctly from the partially-true verdict and routed to cloud-overflow.

**Gaps:**
- No full-HumanEval benchmark of the OURO_NATIVE adaptive Q-exit engine for either accuracy or latency — the entire 'keep the adaptive loop as the default' thesis rests on an unmeasured engine.
- No production rollback / circuit-breaker for the cloud default-tier cutover (only the research spike is gated).
- No specified server-side telemetry to prove adaptive depth is live on the shipped default (the 'mean depth < R4' metric has no wiring).
- No worked cold-start / KV-budget numbers for Ouro-1.4B specifically — reach and interactivity claims rest on generic platform numbers.
- No statement of how the native Q-exit 80-token cap (OURO_NATIVE_MAX) is lifted for real chat answers, or what it costs when lifted.

**Unsupported claims:**
- 'running today's transformers ouro_serve.py container (OURO_NATIVE Q-exit) as the canonical adaptive-depth engine' presented as ship-now/low-risk — the OURO_NATIVE path is ~1s/token, 80-token-capped, and has NO benchmark for accuracy or scaled latency in the repo. Calling it the low-risk default is unsupported by any measurement.
- 'hosted Σ₀ pass@1 within a few points of the local reference (baseline ~0.518)' as the parity gate for the adaptive engine — 0.518 is the fast-cached fixed-depth number; using it as the native-engine parity bar assumes the two engines score identically, which is untested.
- Deck: 'Adaptive early-exit confirmed running on the default tier (mean depth < full loop on easy tokens)' as a success metric — no telemetry path from the served engine is specified, and the repo notes mean_depth/canary instrumentation is research-only and not wired into serving.
- Deck: '~23.7 s per prompt (down from 65.8 s)' implicitly attributed to the looped/default model — it is the fast-cached fixed-depth path on a 48-token-cap golden set on an RTX 3070, not a representative full-answer interactive latency for the adaptive default.
- Portfolio: routing gap characterized as a single omission ('OLLAMA_MODEL_CHAIN omits ouro:latest') — repo shows the static chains target multiple dead/non-served model names, so 'Ouro is the everyday coder after a one-line fix' is unsupported.

**Must-fix:**
- ENGINE CONFLATION ON THE BASELINE — the single most serious correctness defect. Both documents cite pass@1 0.518 and 23.7s/prompt as the Sigma0 baseline AND use 0.518 as the parity gate for the proposed DEFAULT engine (native OURO_NATIVE Q-exit transformers container). But repo evidence is unambiguous: the leaderboard entry label 'ouro-final-rerun-full' has engine 'ouro-fast-cached' (the FIXED-DEPTH cached-generate path), n=164, pass@1 0.518, 67.9s/problem. A grep for native/qexit/OURO_NATIVE/ouro-native in leaderboard.jsonl returns ZERO matches. So both the accuracy figure (0.518) and the latency figure (23.7s/prompt, 67.9s/problem) belong to the fast-cached fixed-depth engine, NOT the adaptive Q-exit engine the plan makes canonical. The native path is UNMEASURED for accuracy and at-scale latency. Fix: relabel every baseline figure as 'ouro-fast-cached / fixed-depth'; state explicitly that the adaptive Q-exit default has NO accuracy or scaled-latency measurement yet; and add a Phase-0 sub-task to benchmark OURO_NATIVE on the full HumanEval set BEFORE asserting it can be the default or that 0.518 is its parity bar.
- NATIVE Q-EXIT LATENCY MATH IS NON-CREDIBLE AS A 'DEFAULT'. ouro_serve.py documents the native path at ~1 s/token, capped at OURO_NATIVE_MAX=80 tokens. That implies ~80s for a single 80-token answer, and the cap itself means the canonical 'deep' default CANNOT emit a long answer at all. The documents present the cloud-transformers-with-Q-exit tier as the ship-now interactive default for 100% of Windows users while quietly inheriting this ~1s/token + 80-token ceiling. Reconcile: either (a) the default is actually the fast-cached fixed-depth path (in which case the 'we never lose the loop / adaptive depth is the default' thesis collapses — vLLM-equivalent), or (b) the default really is Q-exit and the latency/length budget must be stated honestly (80s, ≤80 tokens) and shown to be interactive. As written the deck's 'interactive time-to-first-token' success metric is incompatible with the 1s/token native engine it designates as canonical.
- SUCCESS METRIC 'mean recurrent depth < R4 on easy tokens' IS UNVERIFIABLE ON THE SHIPPED DEFAULT IF THAT DEFAULT IS RUN VIA THE CLOUD CONTAINER WITHOUT INSTRUMENTATION, AND IS DEFINITIONALLY FALSE ON ANY vLLM/fast-cached path. The doc lists this as a default-tier metric but never specifies how mean-depth telemetry is captured server-side, and the ground-truth extraction notes the served deep path uses default qexit while the converge/loop_lm telemetry (mean_depth, canary) is research-only and 'not wired into serving.' Fix: specify the instrumentation that emits mean-depth from the served engine, or drop the metric — do not claim adaptive depth is 'verifiably alive' without a wired telemetry path.
- ROUTING-GAP UNDERSTATEMENT. The portfolio doc frames the issue as 'OLLAMA_MODEL_CHAIN omits ouro:latest.' The actual repo (stream-chat.js :1273-1302) is worse: every static chain (csf/coding/creative/default) lists DEAD model names — 'lantern-csf-dream', 'qwen2.5-coder', 'mistral', a Satyr GGUF — none of which ouro_serve.py serves (its _SERVED_MODELS set does include lantern-csf-dream as a legacy alias, but qwen2.5-coder/mistral/Satyr fall through to cloud). This means the coding intent currently tries qwen2.5-coder FIRST and Ouro never runs for coding unless leaderboard reordering happens to surface it. Phase 0's 'fix the routing gap' is therefore larger than a one-line omission and should be scoped as 'retarget the static chains onto served Ouro aliases,' or the ship-now claim that Sigma0 is the everyday coder is unsupported.
- NO ROLLBACK STORY FOR THE DEFAULT-TIER CUTOVER. The plan has GO/NO-GO gates for the Phase-3 spike but no rollback for the Phase-0/1 cloud cutover itself: if the hosted endpoint degrades, errors, or drifts in production AFTER becoming the chat default, what is the automatic fallback? Memory note 'client-feature-needs-deploy-and-cachebust' and 'surface errors not blanks' are directly relevant. Specify a circuit-breaker: on hosted 5xx/timeout, fall back to local ouro_serve.py on :11434 or the existing cloud-model chain, with a user-visible error, not a blank stream.

**Nice-to-have:**
- State the cloud cold-start numbers as ranges with provenance and a worked example for THIS model. The deck cites RunPod 6-12s uncached / sub-200ms FlashBoot and $0.17-1.10/GPU-hr, but never multiplies through: a 1.4B fp16 load (~2.8GB weights) + container pull is the cold-start driver, and for a single-user chat with no shared prefix vLLM's APC gives little benefit (per the vLLM finding). A one-line 'expected cold start for Ouro-1.4B = X' grounds the 'interactive for payers' claim.
- Quantify the 8GB-fit claim instead of asserting it. The synthesis says weights ~2.8GB fp16 fit 8GB, but the native Q-exit path and the R4 KV (4*24 distinct attention layers per the vLLM ouro.py finding) were flagged as a possible OOM driver at long context. Give the chat-length KV budget number so '8GB dev box runs it' is shown, not assumed.
- The deck's '1.4B reasons like 4B / 2.6B like 8B' should carry the verdict's qualifier that these are FIRST-PARTY, un-replicated benchmark claims strongest on reasoning/math, NOT knowledge recall. A stakeholder deck overstating a vendor benchmark as settled fact is a credibility risk if later challenged.
- Add an explicit decision-cost to Open Question #5 (upgrade base to Ouro-2.6B). Since param-sharing keeps weights light, 2.6B (~5.2GB fp16) still fits 8GB and matches ~8B class — this is arguably a higher-leverage lever than the ONNX spike and deserves a sentence on why 1.4B remains the pinned base (iGPU floor / export tractability) rather than being left as an open question.
- Name the DirectML 'maintenance mode / legacy' fact (conf 0.80, in the appendix) in the risk narrative of the deck, not just the portfolio appendix. Betting the embedded ceiling on a runtime Microsoft has frozen is a material strategic risk the funding stakeholder should see on the risk slide, with the EP-agnostic-export mitigation.

## Lens: Stakeholder/exec readiness of the DECK — single ask, cost, timeline, risk, success metric, jargon control, and pre-emption of skeptical pushback.
- **Grade:** B+  ·  **Staff-level:** True  ·  **Verdict:** revise

**Strengths:**
- Strong central narrative that genuinely de-risks the decision: the 'cloud-now floor + embedded ceiling, one swappable contract' framing lets the exec say yes without betting the product on unproven research. This is the right shape for a funding ask and is communicated clearly.
- Jargon is mostly well-controlled for a non-ML reader: 'thinking in a circle,' 'depth-by-repetition not depth-by-size,' and the memory-light intuition ('same ~2.8GB no matter how many loops') are genuinely good plain-English explanations of an esoteric architecture.
- Confidence scores are surfaced on load-bearing claims and the evidence is primary-source-grounded, which pre-empts the 'is this vibes or verified?' pushback better than most decision decks.
- Intellectual honesty about the killer risk (unprecedented ONNX export, GPU early-exit unverified) and the gating that keeps it off the critical path — an exec can trust a deck that names its own failure mode and shows the fallback.
- The QLoRA-vs-North-Star reconciliation is handled crisply and pre-empts the principled objection that 'you said no retraining.'

**Gaps:**
- No total timeline or total engineering-effort number an exec can sign off on.
- No bounded monthly cloud-cost estimate; warm-replica always-on cost is omitted from the cost slide despite being committed in Phase 1.
- No status-quo / cost-of-inaction baseline to justify the spend against.
- Baseline success metric (0.518) is undefined for the audience and single-row in the source data.

**Unsupported claims:**
- '~$0.17-1.10/GPU-hr, $0 when idle' presented as the cost story while Phase 1 commits to warm/min-1 replicas for paying tiers — the two are in tension and the always-warm cost is never shown, so the implied 'cheap/idle-free' cost is unsupported as a total.
- 'Embedded tier: $0 runtime' is true for cloud compute but glosses the engineering and per-release re-export/requantize maintenance cost the synthesis itself flags — '$0' overstates it.
- '100% of Windows users can reach Σ₀' for the embedded path specifically depends on DX12-capable GPU + the unverified DirectML GPU early-exit; stated as a flat fact on success-metric slides without the embedded caveat that the body acknowledges.
- Success metric 'accuracy within a few points of local reference (pass@1 ~0.518)' implies 0.518 is a stable baseline; the leaderboard contains only one row at that value alongside several 0.0 rows, so 'reference' is weakly substantiated as written.

**Must-fix:**
- TIMELINE/COST DON'T SUM TO A NUMBER THE EXEC CAN APPROVE. The deck says '~2-3 weeks' (Phases 0-1) and '1-2 weeks' (Phase 3) but never states total elapsed time, total engineering weeks, or where Phase 2 and Phase 4 land on the calendar. An exec approving a plan needs one line: 'X weeks of work to ship; embedded spike is a separate Y-week bet; decision point at week Z.' Add a single timeline strip across all phases with the Phase 3 GO/NO-GO date called out.
- THE COST SLIDE HIDES THE REAL RECURRING BILL. '$0.17-1.10/GPU-hr, $0 when idle' is the per-unit price, not the monthly exposure. The plan also commits to WARM/min-1 replicas for paying tiers (Phase 1) which explicitly defeats scale-to-zero — so the floor cost is NOT $0. An exec will immediately ask 'what's my worst-case monthly cloud bill at N users?' Give a bounded monthly estimate (e.g. low/expected/high at assumed user counts) and state the warm-replica always-on cost separately. Right now cost reads as 'basically free,' which the body of the synthesis contradicts.
- SINGLE ASK IS ACTUALLY FOUR ASKS. The 'Ask, In One Line' slide promises one decision, but the closing slide lists four asks (approve, rule on adaptive-exit, accept cloud dependency, confirm QLoRA). Asks #2 and #4 are internal/technical judgments a non-ML exec is not equipped to 'rule' on and shouldn't be on the critical path of a funding decision. Demote #2 and #4 to 'FYI / my recommendation, flag if you disagree' and keep the exec ask to TWO real decisions: fund the plan (#1) and accept the temporary cloud dependency + its cost (#3).
- SUCCESS METRIC 'pass@1 ~0.518' IS UNDEFINED FOR THE AUDIENCE AND THINLY SOURCED. A non-ML exec has no idea whether 0.518 is good or bad, and the leaderboard file actually contains multiple rows at pass@1 0.0 and only a single 0.518 row — so the baseline reads as fragile/cherry-picked without context. State it as a relative, plain-English bar: 'hosted/embedded must score within a few points of what the model scores on the dev box today (our internal coding benchmark),' and drop the bare number from exec-facing slides or footnote it. As written it invites the pushback 'is this one good run or a stable baseline?'
- NO 'WHY NOW / COST OF DOING NOTHING' AND NO 'WHAT IF WE DON'T FUND THE SPIKE.' The deck argues the plan well but never frames the status-quo cost (Σ₀ today can't serve as the everyday coder, falls through to paid cloud models — that IS a recurring cost and a product gap). A skeptical exec asks 'why not just keep using the cloud models we already fall back to, and skip Ouro serving entirely?' Pre-empt this explicitly: state what the project loses (local-first differentiator, the consolidation, cost of third-party API calls) if we DON'T invest, so the spend has a baseline to beat.

**Nice-to-have:**
- Add a one-glance decision tree on the recommendation slide: 'Phase 3 GO -> embedded default, cloud overflow' vs 'Phase 3 NO-GO -> cloud stays the product.' The branching outcome is the most reassuring thing for an exec (the bet can't sink the ship) but it's buried in speaker notes rather than shown.
- The 'deep vs fast' user-facing selector is presented as a clean feature, but an exec/PM may push back that exposing an inference-internals toggle to 'normie' users contradicts the minimize-friction brand. Either justify it as a power-user affordance with a sensible default, or cut it from the exec deck and keep it as an implementation detail.
- Quantify the reach claim. 'Reaches 100% of Windows users' is asserted repeatedly; back it once with the mechanism in one clause (any device with a browser/HTTPS for cloud; any DX12 GPU for embedded) so it doesn't read as marketing.
- The DirectML 'maintenance mode / legacy' risk (the runtime the whole embedded bet rides on is officially being sunset in favor of Windows ML) appears in the synthesis but NOT in the deck's risk slide. An exec who later learns the chosen runtime is deprecated will feel it was hidden. Add one line: 'embedded runtime is mature-but-frozen; we keep the export portable to its successor.'
- Slide count/pacing: 18 slides for a single funding ask is on the long side for an exec. Consider merging 'What Ouro Is' + 'Why best for Σ₀' and 'Options' + 'What the evidence said' to tighten to ~12-13.
- Consider stating the engineering opportunity cost in solo-dev terms: this is a solo developer, so '1-2 weeks specialist time' really means 1-2 weeks where nothing else ships. Naming that makes the spend honest and the ask more credible.

## Lens: Keystone CLAUDE.md North Star — one loop / four objects; models-replaceable; LOCAL-FIRST; no architectural sprawl; the "Σ₀=Ouro, do not rebuild a dense/Qwen track" consolidation; and the "persistent learning, NOT weight modification" principle.
- **Grade:** A-  ·  **Staff-level:** True  ·  **Verdict:** ship

**Strengths:**
- Consolidation honored hard: 'one model (Ouro), one client contract, engines swappable by config' is the explicit organizing principle. The plan never proposes a second/dense coder track — Option 2 (dense fallback) and the GGUF-unroll shortcut are both evaluated AND explicitly rejected precisely because they 'drop the loop,' and Phase 3's NO-GO rule states 'do NOT fall back to a loop-dropping dense GGUF.' This is exactly the consolidation discipline CLAUDE.md demands.
- Models-replaceable is structurally enforced, not just asserted: the SIGMA0_BASE_URL/SIGMA0_API_KEY indirection turns the engine into config; the chat agent 'never branches per backend.' This is the cleanest possible expression of the 'Convergence Core never assumes a specific LLM / all models plug in as replacements' rule.
- The QLoRA-vs-'persistent learning, not weight modification' tension is NOT swept under the rug — it is named explicitly in a dedicated section in BOTH artifacts and reconciled with a defensible argument: the adapter is a one-time, human-triggered, eval-gated FORMAT/tool-call alignment frozen as a static base artifact ('which model plugs in'), while experiential learning still flows through JSONL memory + CSF + Convergence Records via retrieval. The running agent does not retrain on user interactions. This is the correct reading of the North Star and the single most important thing a reviewer needed to check.
- Adaptive Q-exit (the loop's signature property and the whole point of Σ₀=Ouro) is treated as load-bearing and defended: the canonical default is the transformers OURO_NATIVE path that preserves it; vLLM is explicitly demoted to opt-in 'fast'/fixed-R BECAUSE it pins total_ut_steps. A weaker artifact would have made vLLM the default for speed and silently collapsed Σ₀ to a fixed-depth loop — the deck even calls this out as a risk.
- Ties cleanly to the one loop: framed as the 'Reason -> Act' engine, four-objects vocabulary (Memory/Convergence Record) is used correctly, and no new top-level subsystem is introduced — it extends the existing chat path rather than adding sprawl.
- Evidence discipline is genuinely staff-level: every load-bearing claim carries [claim, verdict, confidence, source], confirmed vs refuted vs partially-true are distinguished, and the synthesis confidence (0.82) is honestly lower than the floor's because the embedded ceiling rests on unprecedented engineering. This matches the 'nothing accepted without evidence' rule. I independently verified two repo claims: the OLLAMA_MODEL_CHAIN omitting ouro:latest (real, stream-chat.js:1273-1302) and the QLoRA template/script match (train-qlora-ouro.py) — both check out.

**Gaps:**
- Neither artifact addresses what happens to the existing local Ollama-chain models (qwen2.5-coder, mistral, lantern-csf-dream) that are STILL in the live default chain. Are they being removed as part of the consolidation, or do they coexist? If they coexist, the 'one coder' claim is aspirational, not current-state. This should be explicit.
- No measured number anywhere for the cloud-transformers tier latency — the baseline 23.7s/prompt is the LOCAL RTX 3070 figure; the artifacts assume cloud GPU is faster but never state a target tok/s for the shipped default, only 'acceptable for interactive chat.' For a chat product where the current pain IS latency, the shipped floor needs a latency exit-criterion, not just a parity (accuracy) one.
- The decision asks the stakeholder to rule whether adaptive-depth is non-negotiable (decision_ask #2) but the rest of the plan is already architected as if the answer is 'yes' (transformers default everywhere). If the stakeholder says 'fixed-R4 is fine,' large parts of the rationale (and the entire justification for the slower transformers default and the Phase-3 Variant-B adaptive export) collapse — the plan should sketch that alternate branch rather than presenting one path.

**Unsupported claims:**
- 'Hosted transformers container preserves adaptive depth exactly' — true that the CODE is identical, but the artifacts have NOT verified the hosted/containerized path reproduces OURO_NATIVE outputs vs local (Phase 0 sets this as an exit criterion, so it is correctly an unvalidated assumption, but the prose states it as fact in places — it should be consistently framed as 'expected, gated by Phase 0 parity').
- '~$0.17-1.10/GPU-hr, $0 when idle' for the cloud tier is presented without accounting for cold-start frequency or the warm/min-1-replica tier that the SAME plan says paying users need — a min-1 warm replica is explicitly NOT $0-when-idle. The cost slide and the auth/tier phase contradict each other slightly; the 'cheap, scale-to-zero' framing undersells the warm-replica cost the plan itself requires.
- 'reasons like a 4B model (2.6B variant ~ 8B)' in the deck — these are first-party (ByteDance) benchmark numbers with no independent replication (the validation research itself flagged this). The deck states them as established fact without the 'vendor-reported, not independently replicated' caveat that the underlying research correctly attached.

**Must-fix:**
- LOCAL-FIRST inversion must be named as a real deviation, not soft-pedaled. CLAUDE.md lists local ownership as a non-negotiable feature, yet the RECOMMENDED ARCHITECTURE NAME is 'EMBEDDED-FIRST' while the SHIPPED DEFAULT for ~2-3 weeks (and indefinitely if Phase 3 is NO-GO) is a CLOUD tier where 'user prompts leave the machine.' The label and the reality diverge. Fix: either rename the shipped state honestly (e.g. 'Cloud-default today, embedded-first target') or add an explicit, time-boxed commitment that the cloud-default is temporary with a NO-GO contingency that still preserves a local path (the existing ouro_serve.py NVIDIA-local tier already exists — make it the local-first floor for NVIDIA users so 'local-first' is never fully abandoned, not just an afterthought tier).
- State the NO-GO local-first outcome concretely. If Phase 3 fails, the artifacts leave the product as cloud-default with no universal local option — that is a standing violation of the local-first principle, not a temporary one. The decision_ask #3 ('accept the cloud dependency... explicitly temporary IF the embedded spike lands') quietly admits this but does not force the stakeholder to confront the permanent-cloud-default branch. Add an explicit decision point: if NO-GO, is permanent cloud-default acceptable, or is a fixed-R local embedded (which the spike proves first) the mandated local floor?
- Resolve the verified routing gap as a tracked deliverable, not a parenthetical. stream-chat.js:1273-1302 confirms OLLAMA_MODEL_CHAIN omits ouro:latest entirely — the chains list lantern-csf-dream/qwen2.5-coder/mistral. This means the 'one coder = Ouro' consolidation is NOT actually wired into the live default chat path today; Σ₀ is reinjected via OLLAMA_MODEL or falls through to cloud/Qwen. Phase 0 mentions fixing it, but given it directly undercuts the consolidation claim, it should be the FIRST exit-criterion, with a test asserting ouro:latest is reachable on the default intent — otherwise the artifacts overstate how consolidated the current system is.

**Nice-to-have:**
- The QLoRA reconciliation, while correct, would be stronger with a hard governance guardrail in the artifact: state that the adapter corpus must NOT include user-session experiential data (only curated format/tool-call examples), so there is a bright line preventing future drift from 'format alignment' into 'de-facto online learning on user interactions.' Right now the principle is honored by intent; codify it as a rule.
- vLLM's 'fast' tier still computes the full param-shared loop at fixed R4 — clarify for the stakeholder that this is NOT a dense model and does NOT violate the consolidation (it keeps the loop, just non-adaptively). The deck risks a reader conflating 'fixed-depth' with 'dense.' One sentence prevents that misread.
- The 2.6B-upgrade open question is good, but tie it explicitly to models-replaceable: since the contract makes the engine swappable, note that a 1.4B->2.6B base swap is a config/eval-gate change, not an architecture change — reinforcing the North Star payoff.
- Add a CSF/Memory hook: the artifacts mention Convergence Records abstractly but never specify that engine-parity eval runs (the leaderboard gate) should themselves be logged as Convergence Records [hypothesis/evidence/result/confidence]. That would make the serving decision itself an instance of the loop it serves.
- Consider stating the long-context KV ~4x-at-R4 routing rule as a concrete threshold (token count) that triggers cloud-overflow, rather than 'long-context jobs route to cloud' — makes the Verify/Act boundary testable.
