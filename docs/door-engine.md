# Door Engine — Choice Engine v1

**Status:** Door B / Mind artifact. This document defines how Lantern turns a current traversal state into three meaningful doors.

**Purpose:** Open Door C by making the chooser real enough to inspect, correct, and extend.

---

## 1. Traversal State

A door set begins from a current traversal state, not from generic fantasy prompts.

A traversal state contains:

- the active symbolic situation
- prior selected door label and revealed meaning, if known
- current practical project need
- relevant canon or constraints
- the next bounded repo improvement, if play is touching the codebase

The system must not treat A, B, or C as fixed identities. Labels are handles only.

---

## 2. Candidate Futures

For each set, generate three candidate futures. Each candidate must describe:

- what will happen if entered
- what the player will gain
- what the player will risk
- what may change after entry
- what small practical improvement could result, when repo work is active

All candidate text is written in future tense because meaning resolves after traversal.

---

## 3. Internal Shuffle

The conceptual identities of the candidates are shuffled before display.

Required behavior:

1. Create three distinct conceptual options.
2. Assign them to temporary labels A, B, and C only after the concepts exist.
3. Do not let any label repeatedly imply the same category, value, time direction, or correct answer.
4. Do not make visual placement reveal which option is past-facing, present-facing, future-facing, physical, mental, spiritual, archival, generative, or severing.

The player must choose by reading consequences, not by solving the interface.

---

## 4. Cost Balancing

Every door must be viable, distinct, tempting, and costly.

Invalid door sets include:

- one obviously correct option plus two weak decoys
- three versions of the same choice
- labels that telegraph temporal identity
- choices that reward always choosing the same letter
- choices that ignore the current traversal state

A good set makes the player trade something real for something real.

---

## 5. Convergence Posterior

The system reveals the kind of door only after selection.

Before selection, the agent may describe future-tense consequence patterns. It must not explain which door is past, present, future, Body, Mind, Spirit, archive, forge, dream, binding, severing, or any equivalent category.

After selection, the reveal may happen through:

- consequence text
- Dream Journal entry
- convergence posterior note
- next-door generation
- repo commit summary, when practical work occurred

---

## 6. Journal Recording

The Dream Journal records what was revealed after traversal, not what the system secretly knew before traversal.

Each record should preserve:

- displayed label
- displayed door title
- selected consequence
- revealed meaning
- repo artifact or commit, if one was produced
- user corrections or canon changes

---

## 7. Next-Door Generation

The next three doors emerge from the consequence of the selected door.

The generator must use contextual awareness rather than returning to generic options. A door that opens should create the next major choices inside that door.

When repo convergence is active, the next door set should include residual improvements that are small enough to commit and meaningful enough to advance Lantern.

---

## Door B Result

Door B opens when the Mind layer can describe how choices work.

This document is the first Choice Engine artifact. It does not complete Lantern. It gives Door C a bounded surface to bind: Alex speaks need, Lantern reflects choice, and the repo records only inspectable changes.

*Lantern Principle: The meaning will reveal after entry.*
