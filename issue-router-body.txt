## Summary

Dream Chat should act as the **orchestration router / front door** for Lantern OS. It classifies user intent, selects the correct agent/surface, invokes !convergence loop, waits for results, and surfaces the final response. Dream Chat itself should not be RP mode or Engineering Mode.

## Architecture

**Current (Incorrect):**
User input → Dream Chat switches mode (RP / Engineering) → direct output

**Desired (Correct):**
User input → Dream Chat classifies intent → routing decision → delegate via convergence → wait → surface result

## Route Decision Model

Each user request produces a route object:

```
{
  intent: "code" | "rp_game" | "strategy" | "memory_export" | "dream_analysis" | "trading" | "unknown",
  agent: "keystone" | "founder" | "lantern" | "blinkbug" | "trading" | "csf" | "three_doors",
  surface: "dream_chat" | "three_doors" | "convergence" | "csf_export",
  confidence: number (0.0-1.0),
  reason: string,
  requires_convergence: boolean
}
```

## Routing Examples

| User Input | Intent | Agent | Surface | Reason |
|-----------|--------|-------|---------|--------|
| "make changes to the repo" | code | keystone | convergence | code/implementation request |
| "let's play three doors" | rp_game | three_doors | three_doors | explicit game request |
| "help me think through this strategy" | strategy | founder | convergence | planning/future-focused |
| "export my dreams to archive" | memory_export | csf | csf_export | CSF/CADD data operation |
| "what do my recent dreams mean?" | dream_analysis | lantern | dream_chat | symbolic/reflection request |
| "check my trading signals" | trading | trading | convergence | market/position request |
| "hey, what's up?" | unknown | lantern | dream_chat | ambiguous: default to dream |

## Agent Capability Registry

Each agent should declare:
- id: string
- name: string (display name)
- intents: string[] (which intents it handles)
- triggers: string[] (example keywords that trigger it)
- input_contract: object (what format it expects)
- output_contract: object (what it returns)
- converges: boolean (can run through !convergence loop)
- blocking: boolean (UI should wait for full completion)

Example:

```json
{
  "id": "keystone",
  "name": "Keystone (Code Coordinator)",
  "intents": ["code", "refactor", "bug_fix"],
  "triggers": ["make changes", "integrate", "add tests", "implement", "deploy"],
  "input_contract": { "message": "string", "context": "object" },
  "output_contract": { "plan": "string", "files": "string[]", "changes": "string" },
  "converges": true,
  "blocking": true
}
```

## UI/UX Wait State

When routing through convergence, show progress:

```
User: "integrate the trading agent into dream chat"

[Routing card appears]
Intent: code
Agent: Keystone via convergence loop
Status: queued

[Spinner]
"Routing through convergence…"
"Keystone is handling this…"
"Waiting for result…"

[On completion]
[Result from Keystone displays]
```

Do NOT silently block. Always show routing decision and waiting status.

## Key Decisions

1. **Dream Chat is a router, not a mode switcher.** It should not become engineer, RP, or any other persona. It routes to agents that embody those roles.

2. **Intent classification should be deterministic first.** Use keyword/rule-based routing initially, but structure for embedding/model-based classification later.

3. **Convergence layer is the delegation mechanism.** Do not duplicate convergence logic inside Dream Chat. Call the existing implementation.

4. **Existing work from commit 574da1a is preserved.** Engineering triggers, engineer persona, structured output format — these remain, but move behind routing rather than inside Dream Chat.

5. **Three Doors and RP agents remain.** They are not removed; they become route targets.
